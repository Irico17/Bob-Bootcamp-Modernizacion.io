Descripción de la app

JOSSO Gateway es un servidor de Single Sign-On que permite a los usuarios autenticarse una sola vez y acceder a múltiples aplicaciones sin volver a ingresar sus credenciales. Incluye un panel de administración para monitorear en tiempo real el estado del servidor, la salud de los componentes y las librerías cargadas. Está diseñado para correr en WebLogic con Java 8 y se gestiona desde línea de comandos con Podman y Make.

Objetivo del laboratorio

Migrar JOSSO Gateway de WebLogic 12.2.1.4 a WAS Liberty sin cambiar la lógica de negocio: adaptar configuración del servidor, contenedor y puntos donde el código asume WebLogic.

El workflow de Java Modernization de Bob guía paso a paso el proceso de mover una aplicación Java de un servidor legacy a Liberty.

Prompts post-migración (Liberty end-to-end)

Cuando el workflow termine, la migración debería verse en Podman. Para el panel admin de Liberty (como en WebLogic), usa estos prompts:

Primer prompt

Bob, dame las urls activas y funcionales de la aplicación

Segundo prompt

Configura el despliegue de JOSSO Gateway en Liberty para que funcione correctamente end-to-end. Aplica todos estos cambios de una sola vez:

1. gateway/src/main/liberty/config/server.xml
Features: servlet-3.1, jsp-2.3, mpMetrics-1.1, adminCenter-1.0, localConnector-1.0
Los puertos NO deben ser números fijos — deben ser variables de Liberty: httpPort="${liberty.http.port}" y httpsPort="${liberty.https.port}"
El elemento <application> debe tener: location="josso-gateway-web-1.8.12-SNAPSHOT.war", context-root="/josso-gateway-web", id="josso-gateway-web", name="josso-gateway-web"
Las credenciales deben usar ${liberty.admin.user} y ${liberty.admin.password} en <quickStartSecurity> (NO usar ${env.*}, no funciona en quickStartSecurity)

2. Containerfile (raíz del proyecto)
Las rutas de copia deben ser relativas a la raíz del proyecto: ./gateway/src/main/liberty/config/* y ./gateway/web/target/*.*ar y ./gateway/src/main/liberty/lib

3. Makefile — variables y target liberty-setup
Los puertos deben definirse con ?= para poder sobreescribirse: LIBERTY_HTTP ?= 9086 y LIBERTY_HTTPS ?= 9449
Antes del podman run, generar $(HOME)/josso-bootstrap.properties con printf que incluya las 4 propiedades: liberty.admin.user=admin, liberty.admin.password=admin, liberty.http.port=$(LIBERTY_HTTP), liberty.https.port=$(LIBERTY_HTTPS)
El podman run debe montar ese archivo: -v $(HOME)/josso-bootstrap.properties:/config/bootstrap.properties:ro
No usar -e LIBERTY_ADMIN_USER ni -e LIBERTY_ADMIN_PASSWORD

4. gateway/web/src/main/webapp/admin/libraries.jsp
Reemplazar el bloque de detección del servidor (variables serverInfo, isWebLogic, isLiberty, adminUrl, etc.) por <%@ include file="/WEB-INF/jsp/_server-detect.jsp" %> seguido de String adminLabel = isWebLogic ? "WLS Console" : (isLiberty ? "Liberty Admin" : null);
Buscar y eliminar cualquier línea String ctx = request.getContextPath(); que quede en el mismo archivo después del include — ya viene declarada dentro del include y causaría error de compilación JSP Duplicate local variable ctx

Una vez aplicados todos los cambios, ejecuta make liberty-setup y confirma que:
Liberty arranca con CWWKZ0001I: Application josso-gateway-web started
No hay errores de autenticación CWWKS1100A
curl -o /dev/null -w "%{http_code}" http://localhost:$(LIBERTY_HTTP)/josso-gateway-web/admin/libraries.jsp devuelve 200

Al terminar, muéstrame las URLs y credenciales de acceso:
Admin Center Liberty (usuario y contraseña)
JOSSO Login
JOSSO Status
Con esto tendrás el panel funcional. Si además quieres el botón Admin desde la app, continúa con los prompts 3 y 4.

Tercer prompt

Revisa los JARs marcados como CRIT o WARN en el Library Stack de health.jsp y elimina los que ya no sean necesarios en Liberty, sin romper la funcionalidad de la aplicación. Justifica cada eliminación y confirma que Liberty sigue arrancando correctamente después de la limpieza.

Cuarto prompt

Ahora haz que el botón Liberty Admin de la barra de navegación use la adminUrl que ya resuelve ServerDetectListener, para que abra el Admin Center de Liberty correctamente desde la aplicación.

Resultado: aplicación desplegada en Liberty, funcional como en WebLogic.
